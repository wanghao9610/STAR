# 代码勘察清单

在起草 EXEC_PLAN(Step 2)前执行。目标:在计划改动 `${CODE_NAME}/` 之前,先弄清它现在有什么。绝不假设某个模块、入口或配置存在——去读它。

1. **解析运行环境**。读 `.env`;取 `CODE_NAME`、`CONDA_HOME`、`PYTHON_HOME`。若 `.env` 缺失,从 `.env.example` 创建,并请用户在跑任何东西前先填机器相关值——不要猜路径(CLAUDE.md §6)。

2. **摸清代码根**。列出 `${CODE_NAME}/`。若只有 `.gitkeep`(空),声明 **greenfield**:计划要从零搭骨架,差距清单就是"全部"。

3. **定位 §2 指名的东西**。对子计划 §2 列出的每个依赖——`${CODE_NAME}/` 下的模块/入口、`datas/` 下的数据集、`inits/` 下的权重——确认它确实存在,并记下真实路径。硬依赖缺失是 Step 1 的 blocker,不是能凭空造出来的东西。

4. **把每个 §3 步骤追溯到代码**。对子计划 §3 任务分解里的每一步,判断:做它的代码是**已存在**、需**修改**、还是需**新建**?这个映射就是喂给 EXEC_PLAN"现状勘察"与"动作清单"的差距清单。

5. **找到运行入口**。弄清项目实际怎么跑——训练/评测入口、配置格式、CLI 约定。读 `execs/run.sh`(项目规范的运行入口)、列出 `execs/scpts/`(运行脚本的存放处);这二者定义了启动约定。后续命令必须走这个入口 + conda 环境,而不是临时的一次性脚本。若 `execs/run.sh` 为空(greenfield),计划可以搭建它;本轮准备的任何可复用启动脚本放到 `execs/scpts/<run>.sh`——而不是散落在 `wkdrs/` 里。

6. **检查上游产物**。对叶子 `depends_on` 里的每个前缀(与 §2 一致),确认上游兄弟已 `exec_status: done` 且其产物确实存在于 `wkdrs/` / `inits/` / `datas/`。若某上游叶子不是 `done`,当前叶子就被 block——说明并停下。

**本步产出**:一份差距清单(每个 §3 步骤:已存在 / 修改 / 新建,附真实路径)+ 已识别的运行入口。它直接喂给 Step 3 的 EXEC_PLAN。
