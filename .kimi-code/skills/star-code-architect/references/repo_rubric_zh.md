# 仓库检索与评分表

分支 A 如何把一份研究计划变成参考实现入围名单，以及一个站得住脚的选择。

## 1. 构建检索要素

从计划里提取（不要凭空编造）：

- **任务领域**——来自 §1（如 open-vocabulary detection、时间序列预测）。
- **方法关键词**——来自 §3（架构家族、关键技术、loss/匹配方法名）。
- **点名的 baseline**——来自 §2/§4。这些是金矿：它们的官方仓库通常就是最好的起点候选。
- **框架与版本**——来自 §3/§4（PyTorch/JAX、CUDA 需求），以及 `.env` 能推断出的本机情况。
- **规模约束**——单卡还是多机，计划瞄准的数据集规模。

检索前把要素以短块展示给用户。

## 2. 检索

优先结构化检索，逐级退化：

1. `gh search repos "<关键词>" --language=python --sort=stars --json fullName,stargazersCount,license,updatedAt,description`（更精细的查询用 `gh api search/repositories`）。需要已登录的 `gh`。
2. 网页检索：`"<baseline 名>" official implementation github`、`"<论文标题>" code`、`<任务> <框架> training github`。
3. 两者都不可用时，请用户直接给 URL——不要凭记忆猜仓库名。

入围 5–10 个。直接淘汰：已归档仓库、仅推理/demo 的仓库、awesome 清单、教程类仓库，以及源仓库可用时的 fork。浅读幸存者的 README（`gh api repos/<owner>/<repo>/readme` 或网页），接近决策时再加读 `setup.py`/`pyproject.toml`/`requirements*`。**评估阶段绝不克隆。**

## 3. 评分

| 维度 | 权重 | 高分长什么样 | 低分长什么样 |
|---|---|---|---|
| 计划贴合度 | 30 | 实现了点名的 baseline 或计划的方法家族；任务与 §1 一致 | 同领域不同任务；用起来得大改架构 |
| 完整性 | 20 | 训练 + 评估 + 配置 + 数据管线俱全 | 只有推理 demo；"训练代码即将开源" |
| 许可证 | 15 | MIT / Apache-2.0 / BSD | copyleft、非商用、仅研究，或**没有许可证** |
| 活跃与成熟度 | 15 | 近期有提交、issue 有响应、有 release/tag | 几年没动；issue 区堆满没人答的"训练跑不起来" |
| 代码质量 | 10 | 结构清晰、有测试、有文档、依赖钉版本 | 上帝文件、零测试、乱 vendor |
| 环境匹配 | 10 | 计划的算力和本机 CUDA/Python 都能跑 | 需要用户没有的硬件或工具链 |

## 4. 许可证指引

在门 1 就摆到台面——让用户睁着眼睛做决定：

| 许可证 | 结论 |
|---|---|
| MIT / Apache-2.0 / BSD | 没问题。保留 LICENSE 文件与署名。 |
| GPL / AGPL | 研究可用；提示日后发布衍生代码必须沿用同一许可证。 |
| CC-NC、"仅限研究"、自定义（如某些企业研究许可证） | 强烈提示：先确认预期的发布路径再选。 |
| 无许可证文件 | 默认保留所有权利。未经用户明确接受不得选用。 |

## 5. 呈现入围名单（门 1）

top 3–5，一个候选一个选项，每项带：一句话贴合理由、许可证、stars、最近更新、主要风险（一个短语）。得分最高者放第一位并注明推荐。始终保留逃生选项："都不合适——细化检索 / 从零起步"。留存最终得分；被选中仓库的那一行记入 `codearc.md` §5。
