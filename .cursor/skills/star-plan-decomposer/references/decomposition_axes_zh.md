# 拆分轴

拆分轴指*用什么方式*把父计划切成子计划。选一个（允许混合，但要确认）。用一道题提出这些轴，推荐项放第一位。提出之前先读父计划的 §3（方法）、§4（实验）、§6（里程碑）。

## 三种轴

### 1. 里程碑 / 阶段 —— 默认推荐

沿父计划 §6 的时间线阶段拆。每个里程碑成为一份子计划。

- **何时用**：父计划的里程碑已经写得比较成形（通常如此——coach 会让第一个里程碑是便宜的 MVP，其余从 deadline 倒推）。
- **产出**：时间上有序的子计划链；依赖大多是线性的（每个阶段交接给下一个）。
- **示例**（`0_open-vocab-det-seg`）：`00_mvp-3way-ablation` → `01_core-method-pipeline` → `02_full-experiments` → `03_writing-submission`。

### 2. 组件 / 模块

沿 §3 方法中可分离的部分拆。每个系统模块成为一份子计划。

- **何时用**：方法能拆成可以较独立地构建与测试的模块。
- **产出**：可并行推进的子计划；依赖构成一个小图（共享接口），而非一条线。
- **示例**（`01_core-method-pipeline`）：`010_desc-generation`（LLM 动态描述生成）、`011_set-matching`（多描述集合匹配）、`012_det-seg-heads`（共享检测 + 分割头）。

### 3. claim → 实验

沿 §4 拆：每条 claim 及其对应的实验/消融成为一份子计划。

- **何时用**：贡献主要偏实证，有若干 claim 和消融，各自需要独立的实验管线。
- **产出**：与论文各条 claim 一一对应的子计划——很容易核对"每条 claim 都有实验"。
- **示例**：`00_ablation-dynamic-vs-static`（claim 1：图像条件化动态生成优于静态模板）、`01_ablation-desc-count`（claim 2：集合匹配降方差）、`02_main-lvis-tail`（claim 3：LVIS 罕见类上的收益）。

## 如何选择

- **默认选里程碑**，除非父计划里程碑较弱或用户另有要求——它贴合计划已有的结构，并给出最清晰的执行顺序。
- 当用户接下来真正要做的是*构建系统*、且模块可并行推进时，优先选**组件**。
- 当用户接下来真正要做的是*跑实验*、且风险是实证性的（每条 claim 是否成立？）时，优先选 **claim → 实验**。
- **深度优先于广度：** 在第 2 层按里程碑拆、再对繁重的里程碑（通常是"核心方法"）在第 3 层按组件递归拆，是完全可以的——往往更好。不要试图在单个扁平层里囊括一切。

## 每份子计划的粒度

- 每次拆解目标 **3–7 份**子计划；≤10 是硬上限（命名规则）。少于 3 份通常说明父计划还不需要拆；多于 7 份通常说明一层里藏了两层。
- 每份子计划应负责**一个内聚、可独立检查的块**——有自己的完成判据。若某单元没有清晰的"完成"测试，要么太含糊，要么应并入某个兄弟。
- 明确写出**non-goals**，使相邻子计划不重叠；所有子计划的并集应覆盖父计划的执行，且没有任何一块被两份子计划同时占有。

## 各轴隐含的依赖

把依赖记为每份子计划的 `depends_on` frontmatter 列表（必须先完成的兄弟前缀），并在父计划 `## Sub-plans` 索引里渲染出由此得到的顺序。拆分轴给出默认形状：

- **里程碑 / 阶段 → 线性链**。每个阶段交接给下一个：`01 depends_on ["00"]`、`02 depends_on ["01"]`……执行顺序是一条直线。
- **组件 / 模块 → 小 DAG**。共享接口的模块依赖产出该接口的一方；相互独立的模块 `depends_on: []`，可并行。例如 `012_det-seg-heads depends_on ["010", "011"]`（若它同时消费描述与匹配器）。
- **claim → 实验 → 大多相互独立**。每条 claim 的实验管线通常自成一体（`depends_on: []`）；只有当一个实验确实复用另一个的产出（如主实验需要消融选定的配置）时才加边。

务必保持**无环**。若两个单元看起来互相依赖，它们其实是一个单元——合并它们，或把共享的部分拆成第三个上游子计划、让两者都依赖它。

## 混合拆解

允许混合轴（要明确确认）。最常见的形状是**本层按里程碑拆、再把繁重的里程碑按组件递归**——但当里程碑与 claim 无法干净对齐时，也可以在同一层里混合。混合时说清哪个单元来自哪条轴，并仍给每个单元一个 `depends_on` 列表。

- **示例**（`0_open-vocab-det-seg`，第 2 层混合）：
  - `00_mvp-3way-ablation` —— 来自 **claim** 轴（先便宜地验证核心 claim）；`depends_on: []`
  - `01_core-method-pipeline` —— 来自**里程碑**轴（搭建完整方法）；`depends_on: ["00"]`
  - `02_full-experiments` —— 来自**里程碑**轴（其余全部 claim/消融）；`depends_on: ["01"]`
  - `03_writing-submission` —— 来自**里程碑**轴；`depends_on: ["02"]`
  - 再把 `01` 按**组件**递归成 `010/011/012`（见上面的组件示例）。
  - 执行顺序：`00 → 01 → 02 → 03`，其中 `01` 展开成自己的组件 DAG。
