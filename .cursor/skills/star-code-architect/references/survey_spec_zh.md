# 勘察规范

只读侦察：把代码库变成一份架构设计（Step C1）敢于信赖的**仓库地图**。勘察者绝不修改任何文件。

## 分线

一条关注面一个 `Task` subagent（`subagent_type: explore`），最多 3 个并行（只读，所以并行是安全的）：

| 线 | 看什么 | 关键问题 |
|---|---|---|
| 结构与依赖 | 顶层目录、包布局、内部 import | 有哪些模块？import 朝哪个方向流动？有环吗？ |
| 配置系统 | 配置文件/目录、注册表机制、CLI 参数解析 | 一个实验怎么指定？走注册表还是普通 import？ |
| 数据管线 | dataset 类、transforms、loader、下载脚本 | 数据集从哪接入？有硬编码路径吗？ |
| 训练/评估入口 | train/test 脚本、engine、循环、调度器 | 运行面是什么？单入口还是多入口？ |
| 脚本与工具 | tools/、scripts/、shell 文件、notebook | 哪些是必需的，哪些是一次性杂物？ |
| 测试与文档 | tests/、CI 配置、docs/ | 覆盖了什么？哪些能当快速验证套件用？ |

**轻量模式**（分支 A 的新克隆，或不到约 50 个 Python 文件的仓库）：把各线合并成一次扫描——由一个 `explore` Task subagent 或主循环自己完成。

## 每个勘察者的契约

- **范围**：只看自己这条线，只读；"不得修改、创建或删除任何文件。"
- **返回**（结构化）：
  - `inventory`——本线的目录/模块，每项一行：路径 + 职责。
  - `entrypoints`——发现的可运行入口（脚本、CLI 命令），在范围内时提供。
  - `mechanisms`——这条线怎么运作（配置风格、注册表用法、数据流向），3–6 条要点。
  - `smells`——只报可能促成迁移项的发现：死代码、重复、跨层 import、命名不一致、巨型文件。每条：路径 + 一行证据。
  - `unknowns`——没弄清的事及原因。

## 汇成仓库地图

主循环把各线报告汇成一份仓库地图：

1. **模块清单**——带注释的目录树，每个顶层目录一行职责。
2. **依赖方向**——哪些层 import 哪些层；标出越界之处。
3. **排序坏味道**——跨线去重，按"对计划工作的阻碍程度"排序；只保留值得立迁移项的。其余记为 `codearc.md` §3 的风格备注，不立迁移项。

仓库地图直接喂给 Step C1，对用户只做摘要展示（不整篇倾倒）。
