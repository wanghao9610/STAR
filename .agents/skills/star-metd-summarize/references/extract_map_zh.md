# 提取映射——计划小节 → 方法文档

下文引用的计划 §n 指两套计划模板：**策略计划**（根与中间节点）含 §1 问题定义与动机、§2 相关工作与定位、§3 核心方法、§4 实验与验证设计、§5 风险与退路、§6 里程碑与交付物；**执行子计划**（叶子）含 §1 目标与范围、§2 输入与依赖、§3 任务拆解、§4 交付物与输出、§5 验证与完成判据、§6 局部风险与退路。引用的项目规范 §n 指 AGENTS.md 的编号章节。

## 相关性：哪些叶子喂哪份文档

按叶子的 §2 输入、§3 步骤、§4 交付物**点名**了什么来归类——绝不靠标题猜：

| 叶子里的信号 | 喂给 |
|---|---|
| 一个 `datas/` 路径；下载、过滤、标注、转换或生成数据的步骤 | dataset |
| 一个 `${CODE_NAME}/` 模块；§3 里描述的组件、架构或损失 | framework |
| 一个 `inits/` 权重；训练入口、优化器、schedule 或超参数 | training |
| 一个 benchmark、指标、baseline、评测入口或消融变体 | evaluation |
| 它的 §1 目标——永远：每个叶子都是方法的一个组成部分 | overview |

一个叶子常常同时喂多份文档（既造数据又在其上训练的叶子同时喂 dataset 与 training）。一个信号都不点名的叶子只喂 overview。策略节点按下面的映射逐节喂。

## 逐文档映射

每一行读作*来源 → 它填的文档小节*。来源不存在就是缺口，不是默默跳过。

### overview.md

| 文档小节 | 来源 |
|---|---|
| 1. 问题与动机 | 根 §1 |
| 2. 现有工作的不足 | 根 §2（各自留下的具体短板与本方法的定位；只点作品名——引文在 `metds/refs/`） |
| 3. 核心思想 | 根 §3（关键洞察 + 技术路线；实现细节留给 framework） |
| 4. 方法总览 | 每个叶子的 §1 目标，一行一个；中间节点的 §3（当它归拢了若干组件时） |
| 5. 贡献与 Claims | 根 §3 novelty claims + §4 claim→实验映射 |
| 6. 状态与里程碑 | 根 §6 + 各叶子的 `exec_status`（只写几行——计划树的实时进度view 归 status skill） |

### dataset.md

| 文档小节 | 来源 |
|---|---|
| 1. 数据集清单 | 根 §4 数据集选择 + 每个叶子 §2 的 `datas/` 输入 |
| 2. 逐数据集详情 | 根 §4 的选型理由 + 点名了该数据集的叶子的 §1/§3 |
| 3. 预处理 | 数据相关叶子的 §3 步骤 + 它们产出的 §4 交付物 |
| 4. 自建数据 | 数据构建类叶子的 §1 目标、§3 流水线、§4 产物、§5 质量检查 |
| 5. 统计信息 | 只取计划写明的数字（根 §4、叶子 §2/§4）；需要跑一次才能得到的数字是缺口，而不是"去补跑"的待办 |
| 6. 数据集 → 实验映射 | 根 §4 claim→实验表 + 数据类叶子的 `traces_to` |

### framework.md

| 文档小节 | 来源 |
|---|---|
| 1. 总体架构 | 根 §3 技术路线 |
| 2. 核心组件 | 建模类叶子的 §1 目标、§3 步骤、§4 交付物 + 其 §2/§4 点名的 `${CODE_NAME}/` 路径 |
| 3. 关键设计决策 | 根 §3 "为什么它该work" + 叶子 §6 里能解释某个选择的局部风险；根 §5 Plan B（当它点名了被否掉的替代方案时） |
| 4. 与已有工作的差异 | 根 §2 定位 + §3 novelty type |
| 5. 模块映射 | 各建模叶子的 `${CODE_NAME}/` 路径 + 拥有它们的叶子 |

### training.md

| 文档小节 | 来源 |
|---|---|
| 1. 训练流程总览 | 根 §3 训练策略 + 训练类叶子的 §1/§2（阶段顺序依 `depends_on`，其次按前缀） |
| 2. 逐阶段详情 | 训练类叶子的 §3 步骤 + §2 的 `inits/` 初始化 + 根 §4 计算预算 |
| 3. 关键超参数 | 任何计划写明的值（叶子 §3、根 §4）；没有哪个计划定死的值保持 `TBD` 并列为缺口——绝不用一个看起来合理的默认值填上。已执行叶子敲定的值，应经执行器的 ENRICHED 同步回写进入它的 §3 |
| 4. 训练要点 | 训练类叶子的 §6 局部风险（失败模式、早期信号、退路） |
| 5. 复现方式 | 训练类叶子 §3/§4 里计划记录的入口与命令 |

### evaluation.md

| 文档小节 | 来源 |
|---|---|
| 1. 评测协议总览 | 根 §4（任务、benchmark、指标、baseline） |
| 2. 逐 Benchmark 详情 | 根 §4 的指标与有意义幅度 + 评测类叶子 §5 完成判据的阈值 |
| 3. Baselines | 根 §4 baseline + 评测类叶子 §2（当它点名了某 baseline 的权重或来源时） |
| 4. 消融设计 | 根 §4 消融设计 + 根 §5 kill-criteria（"什么结果推翻它"那一列） |
| 5. 评测执行 | 评测类叶子 §3/§4 的入口、命令与输出位置 |

**分数绝不进 evaluation.md。** 它定义协议；某个 run 测出了什么在该 run 的分析报告里（`wkdrs/<run>/` 下），而这些数字的跨 run 总账是 `wkdrs/results/results.md`（`star-expt-analyst aggregate`）。

## 合并与冲突规则

1. **跨层级去重。** 父计划与其叶子说的同一件事只是一个事实，只写一次，用叶子更具体的说法。
2. **要合并，不要拼接。** 每个小节用一个声音写成，没有逐计划的接缝，也不出现"如计划 01 所述"。读起来像摘录清单的小节就是失败。
3. **冲突：叶子压父计划；`updated` 新的压旧的。** 叶子更具体，且承接 executor 从实际执行同步回来的内容。
4. **无法消解的冲突要打印出来，不要替用户裁决。** 两个同等地位的来源相互矛盾时（同层两个叶子给了不同学习率），两个值就地并列写出、前缀 ⚠、点名两个计划及其小节。绝不悄悄挑一个赢家；绝不取平均。
5. **出处随每段内容一路同行**——`{计划, §, updated, exec_status}`——从提取一直到 `sources:` 块。

## 缺口

没有任何来源覆盖的文档小节保留标题，并写入一行：

```
TODO —— 无计划覆盖；请补充到 <计划文件> §<n>。
```

要点名*应该*承载它的那个计划（策略内容找根计划，执行细节找拥有它的叶子），而不是含糊的"某个计划"。已覆盖小节里单个没写明的值（超参表里的 `TBD`、没说清的 split）用同样一行、点名缺的是哪个字段。缺口只汇报，绝不用一个看起来合理的默认值糊过去。

## "尚未验证"标记

来自 `exec_status` 非 `done` 叶子的段落是设计意图。在承载它的小节末尾加一行斜体：

```
*⚠ 尚未验证——来自 `<计划文件>` §<n>（exec_status: <pending|in_progress|blocked>）。*
```

一个小节一行，不是一句一行；来自 `done` 叶子与策略节点的内容不加标记。小节里两种内容都有时，加标记并只点名未验证的那些来源。

## Frontmatter 契约

每份生成的文档开头是：

```yaml
---
type: <overview|dataset|framework|training|evaluation>
language: <en|zh>
generated: <YYYY-MM-DD>          # 真实日期；绝不编造
sources:                          # 喂过本文档的每个计划，以及读取时它所带的 updated
  - plan: <prefix>_<slug>_plan.md
    updated: <YYYY-MM-DD>
---
```

`type:` + `generated:` 正是标记一份文件"由编译产生、因而可以重新生成"的东西；没有它们的文件是人写的。`sources:` 是过期检查的依据：记录的 `updated` 比该计划当前的 `updated` 旧，就说明文档过期了。

## 提取契约（结构化返回）

收集者只返回下面这两个列表加 `plans_read: <n>`，别的什么都不返回：

```yaml
passages:
  - target_section: "3. 预处理"
    content: "<该事实，改写成能独立成句——不要写'这个计划说'>"
    source: {plan: "00_data-pipeline_plan.md", section: "§3.2", updated: "2026-07-15", exec_status: "done"}
gaps:
  - target_section: "5. 统计信息"
    wanted: "各 split 的样本数"
    suggest: "00_data-pipeline_plan.md §4"
plans_read: 7
```

收集者只做提取。它们不写文件，不做跨计划的冲突解析（两段都返回，裁决是主 agent 的事），不发明计划里没有的事实，也不编译 overview——它需要其余四份文档编译后的内容。

## 变更清单（diff 门）

重新生成一份已存在的编译文档时，写入前按小节各报一行：

| 小节 | 变更 | 变了什么 |
|---|---|---|
| 3. 预处理 | 改写 | 从 `01_..._plan.md` §3.4 新增了 tokenization 步骤 |
| 5. 统计信息 | 未变 | — |

取值为 新增 / 改写 / 删除 / 未变。全部小节都是"未变"的一轮什么都不写——文件与它的 `generated` 日期都保持原样。"删除"行必须点名来源为什么消失了（某个计划小节被删、某个叶子被重新划范围）；没人要求过的删除是"某个计划丢了内容"的信号，即使覆盖已获批准也值得汇报。
