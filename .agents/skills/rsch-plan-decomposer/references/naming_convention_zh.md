# 子计划命名规则

每个计划文件名为 `<prefix>_<slug>_plan.md`。**前缀（prefix）**是一串十进制数字，编码该计划在拆解树中的位置。

## 规则

- **深度 = 前缀长度。** 根计划（由 `rsch-plan-coach` 产出）前缀为 1 位（`0_`）。其子计划前缀 2 位，孙计划 3 位，以此类推。
- **要拆解前缀为 `P` 的计划**，每份子计划的前缀 = `P` **再追加一位数字**，该数字是子计划在兄弟中的 0 起始序号。
  - `0_` → `00_`、`01_`、`02_`、…
  - `00_` → `000_`、`001_`、…
  - `3_` → `30_`、`31_`、…
  - `01_` → `010_`、`011_`、…
- **任一子计划的父级** = 去掉其前缀的最后一位数字（`011` → 父 `01` → 父 `0`）。

## 示例树

```
metds/plans/
  0_open-vocab-det-seg_plan.md        根            前缀 "0"    层级 1
   ├ 00_mvp-3way-ablation_plan.md     "0" 的子 0    前缀 "00"   层级 2
   ├ 01_core-method-pipeline_plan.md  "0" 的子 1    前缀 "01"   层级 2
   │  ├ 010_desc-generation_plan.md   "01" 的子 0   前缀 "010"  层级 3
   │  ├ 011_set-matching_plan.md      "01" 的子 1   前缀 "011"  层级 3
   │  └ 012_det-seg-heads_plan.md     "01" 的子 2   前缀 "012"  层级 3
   ├ 02_full-experiments_plan.md      "0" 的子 2    前缀 "02"   层级 2
   └ 03_writing-submission_plan.md    "0" 的子 3    前缀 "03"   层级 2
```

文件在目录列表中天然有序（`0`、`00`、`01`、`010`、`011`、`012`、`02`、`03`），因此字典序已经反映了树结构。

## 约束与边界情形

1. **每个节点最多 10 个子计划。** 每层只有一位数字，序号只能是 0–9。若某节点看起来需要超过 10 个子计划，**不要**追加第二位数字（`10_` 有歧义——见下）。改为：合并单元，或分两级拆（先拆成 ≤10 个粗单元，再对繁重的递归拆）。这个上限是"该重新划分粒度"的信号，不是需要工程绕过的 bug。

2. **为什么不每层用两位数字。** 前缀 `10_` 可能指"计划 1 的子 0"，也可能指"顶层计划 10"。采用固定的每层一位、且每层节点 ≤10 的方案，前缀才始终无歧义，父级也始终能通过去掉一位数字复原。请保持这个不变式。

3. **前缀是提示；`parent:` 才权威。** `rsch-plan-coach` 目前把每个根都编号为 `0_`，因此两个无关的根计划可能都是 `0_`（仅靠 slug 区分），它们的子计划也都以 `00_` 起始。所以数字前缀供人类排序与提示树结构，但**每份子计划 frontmatter 里的 `parent:` 字段才是真正的链接**。务必把 `parent:` 设为确切的父文件名，并在判断某子计划属于哪个父计划时依赖它（而非前缀）。

4. **slug 与树结构无关。** `<slug>` 是该单元内容的简短、可读的英文名，与父计划的 slug 无关。`0_open-vocab-det-seg` 可以有子计划 `00_mvp-3way-ablation`。

5. **补空位 / 重新编号。** 按确认后的顺序从 0 起密集分配序号。若用户之后删掉中间某个子计划，保留空位而不要重排（重排会破坏所有更深的前缀以及每处 `parent:`/`traces_to` 引用）。若要插入一个，给它下一个空闲序号，即使这样在数字上不连续；排序意图记录在父计划的 `## Sub-plans` 索引里，而非数字本身。
