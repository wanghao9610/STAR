# 归拢 Rubric——提升、落位与发布面打磨

候选文件如何被找到、判定、落位、复核，以及之后的打磨 pass 允许动哪些东西。门槛是刻意抬高的：发布时最大的诱惑就是把整个仓库收拾一遍，而一次说不出理由的提升，只是让审阅者对这份 diff 失去信任的噪音。

## 扫哪里

**扫：**

- `tasks/<plan-name>/`——计划自有的工具脚本（规约 §9 称其为持久文件），以及堆在旁边的各种 scratch。多数真正的候选都在这里。
- `wkdrs/<run>/`——**只扫脚本和复现配置**：某次分析写出的画图脚本、能复现头条 run 的配置。它们通常未被跟踪，因为 `wkdrs/**` 除 `*.md` 外都被 git 忽略——而这恰恰是它们值得提升的原因。
- 项目根——始终没找到归宿的散落 `.py` / `.sh`。
- `execs/scpts/*.sh`——作为**引用对象**读取（README 会打印它们），并检查硬编码路径。它们是启动器，留在 `execs/`（规约 §9），绝不提升进 `${CODE_NAME}/`。

**绝不扫：** `datas/`、`inits/`、`metds/`、`.git/`、`docs/`、任何形式的生成产物（checkpoint、日志、预测结果、图片、`.csv`/`.json` 输出）、`${CODE_NAME}/` 里已有的东西，以及任何超过约 1 MB 的文件（那是披着源码扩展名的产物）。

## 提升检验

候选必须**至少通过一条**才被提升，且通过的证据要写在它那一行上：

- **A. README 会引用它。** `references/readme_map_zh.md` 里的某一节会打印它的路径或运行它的命令——§7 点名的数据准备脚本、§11 打印的评测入口、§9 展示的 demo。
- **B. 某个已执行叶子需要它。** 该文件被某个叶子的 §4 交付物点名，或是其 §5 完成判据所运行的机制，且该叶子 `exec_status` 为 `done`。证据是计划文件与行号。
- **C. 它能复现账本里的数字。** 跑它正是 `metds/results.md` 中某一行的产生方式——账本引用的那次 run 的配置、算出某个已报告指标的脚本。证据是账本的那一行。

一条都不过 → **原地保留**。在该行写明并继续；这是 `tasks/` 里多数文件的预期结果，不是 finding。

有两条否决项凌驾于"通过"之上：

- **含 secret 或机器本地路径** → 保持原样时不提升。它先成为 `release_checklist_zh.md` 的阻断项；等用户清理后再提升。
- **与 `${CODE_NAME}/` 里已有的可用代码重复** → 动作是 `merge` 或 `keep in place`，绝不是再放一份副本。见下。

## 落位

1. 目的地取自 `metds/codearc.md` §2 的放置规则，按文件**是什么**匹配——数据流水线脚本、模型组件、配置、工具、测试。
2. §2 对这类文件没有规则时，查 §4 的计划组件映射，看它实现的是哪个组件。
3. 仍然定不下来 → 该行动作写 `route`，而不是猜一个路径。创建 `codearc.md` 未点名的目录是 `$star-code-architect` 的决定；一次自己造出 `${CODE_NAME}/utils/` 的发布运行，等于没经过 gate 就偷偷修改了架构规范。
4. 根本没有 `codearc.md` → 所有行都是 `route`，整个 gather 阶段报告"架构规范才是阻塞点"。README 的编译仍可继续。

## 重复检测

提升前先与 `${CODE_NAME}/` 比对：相同或近似的函数/类名、可能目录下的同名文件、明显是拷贝过去的函数体。命中时：

- 已有代码更新或等价 → `keep in place`，并注明该候选已被取代。
- 候选实质更好（修了 bug、处理了已有版本漏掉的情况）→ `merge`：该行写明由哪个已有文件接收改动、改什么，且合并是对那个文件的编辑，不是新增文件。
- 尽管相似但确实不同 → `move`，且该行说明二者差别，免得审阅者被两个相似文件搞懵。

## 决策契约

每个候选一个条目，按扫描顺序：

```yaml
- path: <相对项目根的路径>
  kind: <工具脚本 | 配置 | 数据流水线 | 模型组件 | 分析 | 测试 | 其他>
  passes: A | B | C | none
  evidence: <README 的哪一节、计划 file:line、或账本的哪一行>
  destination: <${CODE_NAME}/ 下的路径，或 "—">
  action: move | merge | keep in place | route
  plan_referenced: <本文件移动后会过期的计划 file:line，或 "no">
  risk: <一行——要修的导入方、会变的配置路径、或无风险>
```

`plan_referenced` 不是否决项，而是一个让用户睁着眼批准的后果。计划文本不归本 skill 改，因此每个这样的行都会落进报告的路由块，交给 `$star-plan-reviser`。

## 逐行复核

每批准一行执行完，主循环自己重跑检查——绝不采信自报：

1. 经 `.env` 解释器对目的地跑 `python -m compileall -q <destination>`。
2. 在全仓库跑 `grep -rn "<旧路径>" --include="*.py" --include="*.sh" --include="*.md" .`——任何残留命中要么是还没修的调用点，要么说明这一行没做完。
3. `merge` 的情况：接收方文件能编译，且其原有导入方仍可解析。

尝试修复后仍失败的行要回滚（被跟踪路径用 `git checkout`，其余移回原处），在报告里标 `blocked` 并附失败文本，然后继续处理其余行。

## 发布面打磨

适用范围：本次提升的文件、README 会打印的入口 / 配置 / `execs/scpts/*.sh`、以及 README 会展示的公共 API。仅此而已——`${CODE_NAME}/` 其余部分的六维审计属于 `$star-code-reviewer`，该范围之外的 finding 只记录待路由，不动手修。

这里什么算 finding：

- **移动残渣**——移动留下的孤立 import、仍指向旧位置的路径常量、还在描述文件旧归属的 docstring。
- **随提升带过来的 scratch**——被注释掉的实验变体、`print()` 调试、硬编码的 `if True:` 开关、没用的 `sys.path.append`。
- **`codearc.md` 符合度**——文件名与放置符合 §2 的规则与 §3 的命名约定；§7 残留清单上的名称只标记，绝不重命名。
- **README 点名之物的文档**——读者看完 README 后会去查的每个符号、脚本参数或配置键，都要有说明它做什么的 docstring 或注释。README 从未提及的文件不在范围内，哪怕它就在同一目录。
- **面向读者的清晰度**——README 呈现为一步、实际却身兼数职的函数；与 README 对它的称呼相矛盾的命名。修法取更小的那个改动：通常是改 docstring，有时是改名，绝不是重新设计。

永远不合格的项：改变行为、发布面之外还在使用的签名、发布面之外的文件、改名残留清单上的名称，以及上游继承来的代码（AGENTS.md §3——报告它，别动它）。
