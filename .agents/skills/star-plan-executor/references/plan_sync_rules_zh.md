# 计划同步回写规则 — 让子计划与实际执行保持一致

当执行被证实偏离子计划时,该偏差经用户确认后要**写回子计划**(`metds/plans/<prefix>_<slug>_plan.md`),让用户日后重读计划时看到的就是实际执行的内容。本文件定义:什么算偏差、delta 形式、写回流程,以及**不许**用同步回写的边界。

## 实质性(要同步) vs 细节(不同步)

EXEC_PLAN 本来就*应该*比子计划更具体——"更精确"**不是**偏差,只有一个例外(**ENRICHED**,见下)。只同步**实质性**偏差,以子计划自身的粒度判断:

- 某个 §3 步骤被**新增、删除、替换或重排**;
- 某项 §2 输入/依赖被证实**有误或缺失**(实际用的数据集/权重/模块与写的不同);
- 某项 §4 产出物**路径或形态变了**;
- §5 完成判据被**调整**(阈值、指标或检查方式);
- 子计划未写明的某个值被**执行敲定**,且某份方法文档会引用它(**ENRICHED**,见下)。

不算实质性:一个 §3 步骤内部更细的子步骤;没有任何方法文档会引用的实现选择、命令与路径。拿不准时,当作细节,不动子计划。

## ENRICHED:执行敲定、且文档会引用的值

子计划同时也是方法文档编译的**来源**——`star-metd-summarize` 只读计划、别的什么都不读,所以只存在于 run 日志里的值永远到不了文档。子计划没写明、执行时定了下来、而某个 `metds/*.md` 章节会引用的值,因此不是细节:不回写,它就会在那份文档里变成永久的 `TBD`。

在范围内——该值必须是某个文档章节会引用的,且该行必须点名那个章节:

| 执行敲定的值 | 被谁引用 |
|---|---|
| 超参——lr、schedule、batch size、epochs | `training.md` §3 |
| 实际使用的初始化 / backbone | `training.md` §2 |
| 复现入口——读者会重跑的命令与 config | `training.md` §5、`evaluation.md` §5 |
| 某个组件行为所依赖的关键 config | `framework.md` §2 |
| 实际采用的 split / 预处理选择 | `dataset.md` §3 |

不在范围内:任何文档都不引用的东西——内部 flag、临时路径、seed、环境细节(那些归 EXEC_LOG 与 `freeze.txt`)——以及计划已经写明的值(改动那种值是 MODIFIED,不是 ENRICHED)。**如果你说不出哪个文档章节会引用它,它就是细节。**子计划不是 config 转储。

## Delta 形式

每条偏差一行,类型仿 OpenSpec delta:

- **ADDED** —— 子计划里没有:`ADDED §3.5:"<新步骤>" —— 原因:<…>`
- **MODIFIED** —— 做法与写的不同:`MODIFIED §3.2:"<原文>" → "<新做法>" —— 原因:<…>`
- **REMOVED** —— 写了但不会做:`REMOVED §3.4 —— 原因:<已过时 / 被 … 覆盖>`
- **ENRICHED** —— 子计划没写,执行敲定了它:`ENRICHED §3.2:lr 未写明 → 1e-4 —— 被引用:training.md §3`

规划时发现的行进 EXEC_PLAN 的"与子计划的偏差"表;执行中冒出来的行进 EXEC_LOG 的"待同步修正"。

## 两个同步点,都经用户确认

1. **审批门**。规划时已知的偏差本身就是用户批准的 EXEC_PLAN 的一部分——批准计划即批准同步这些行。批准后、执行前立即写回。
2. **收尾**。执行中冒出的偏差累积在"待同步修正"——绝不为它们中断执行。收尾时**一次性**呈现整批(*全部同步 / 逐条挑 / 不同步*),只写回确认的行。若一轮被弃跑或阻塞,未同步的行留在日志里,下次续跑收尾时再次提出。

## 写回流程

对每条确认的行:

1. **先重读子计划**。若其 §2–§5 在本轮开始后已被改动(用户手改、重新拆解),呈报冲突——不要盲目覆盖。
2. **原地更新受影响的 §2–§5 段落**为确认后的内容;正文读起来必须是当前事实,而不是一块补丁。
3. **追加一条 `## Revision History` 条目**——与 `star-plan-reviser` 写入的是同一个 append-only 小节(文件末尾没有则创建;绝不改写既有条目)。每次同步一个 `###` 块,格式与 reviser 兼容:

   ```markdown
   ## Revision History

   ### <YYYY-MM-DD> — star-plan-executor · <model_id> (run: <prefix>_<slug>, 审批门 | 收尾)
   - MODIFIED §3.2:"<原文>" → "<新做法>" —— <原因>
   - ADDED §3.5:"<新步骤>" —— <原因>
   - ENRICHED §3.3:lr 未写明 → 1e-4 —— 被引用:training.md §3
   ```

4. **更新 frontmatter 的 `updated`**;frontmatter 里其他字段一概不碰。
5. **在来源处标记已同步**(EXEC_PLAN 的 `synced` 列 / EXEC_LOG 的勾选框)。同步回写是幂等的:标记过的行绝不二次应用。

日期来自用户/会话上下文——绝不自造时间戳。

## 边界:永远不同步的东西

- **§1 目标与范围、§6 局部风险**——目标级偏差意味着任务变了,而不是打法变了;那是重新拆解(`star-plan-decomposer`)的事,不是同步回写。
- **任何父计划**——父计划归 `star-plan-coach` / `star-plan-decomposer`(反馈回流)。
- **把 §5 判据放松到与根计划 §4 指标 / §5 kill-criteria 冲突的改动**——那是战略信号:记录它、点明它、走反馈回流。不要同步。
- **测量出来的结果**——ENRICHED 行携带的是执行*选定*的值(lr、backbone、split),绝不是它*测出*的值。分数归 `wkdrs/<run>/EXPT_ANALYSIS_<date>.md`;一份记录自己结果的计划,在所有据它编译的文档里都会被读成设计意图。
- **事后审计与基于证据的修订**——评定一轮执行实际达成了什么、并据证据修订计划(含 §1/§6),是 `star-plan-reviser` 的职责;同步回写只负责在执行过程中让 §2–§5 与经用户确认的执行现实保持一致。

§5 的同步回写必须在 Revision History 条目里明写旧 → 新,让"什么算完成"永远不会悄悄漂移。
