# STOP 线规则 —— Codex 跑什么 vs 用户跑什么

Codex 及其委派的 worker 可以写代码、跑**轻量验证**。任何**重型或不可逆**的动作越过 STOP 线：备好确切命令、写进 EXEC_LOG 的「待用户执行」区，然后停下交给用户。绝不自主越过 STOP 线启动任务。

## Codex 执行（轻量验证）

- 单元 / smoke test、import 检查、极小 batch 的一次前向。
- 小规模、**不微调**的推理,在小子集上——例如 MVP 完成判据:"不训练、小子集、替换文本输入做对比"。
- dry-run、配置校验、shape / dtype 检查、几步 overfit 的 sanity run。
- 任何能在适度资源上几分钟内跑完、生成输出及持久执行 checkpoint 只写入 `wkdrs/<run>/`、中间工作文件只写入 `tasks/<plan-name>/` 的动作。

## 越过 STOP 线 → 交给用户

- **长时或多卡训练 / 微调**——任何完整训练 run。
- **大开销 API 调用**——按次计费的大批量 LLM/VLM 推理(如在全量数据集上生成描述)。
- **全量评测**——耗时数小时或占用可观算力。
- 任何会**覆盖用户可能想保留的已有产物**,或把生成的 run 产物写到 `wkdrs/<run>/` 以外的动作。常规向 `tasks/<plan-name>/` 写入中间工作文件不越过 STOP 线。
- 任何你**无法界定开销或耗时**的动作——拿不准就当 STOP 处理。

## 怎么交接

对每个 STOP 动作,写进 EXEC_LOG 的"待用户执行":

- **确切命令**,走 `.env` 的 conda 环境(绝不用系统 python);存在运行入口时经项目运行入口(`execs/run.sh`)调用;
- **产出什么、存哪**(`wkdrs/<run>/…`);
- **该带回什么输出**以便验证完成判据。

为可复现，Codex 也可以把重命令**写成**一个可运行脚本放到 `execs/scpts/<run>.sh`（写文件是轻量的，跑它不是），让用户一条命令启动。绝不执行它——它仍在 STOP 线之外。

然后停下,在简报里告诉用户哪些命令在等他们。re-invoke 时若这些产出已存在,从完成判据验证处续起(Step 6)。
